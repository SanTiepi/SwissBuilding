# North-Star Metric Tree and Product Instrumentation

Date de controle: `25 mars 2026`

## Purpose

SwissBuilding should not measure success by feature count or document volume.

It should measure whether it is becoming:

- the canonical building workspace
- the easiest procedural path
- the strongest proof reuse layer
- the default operating inbox

## North star

Primary north-star concept:

- `active buildings with recurring operating value`

Meaning:

- a building is not just stored
- it is actively used for workflow, proof, or decisions

## Metric tree

### 1. Canonical truth metrics

Examples:

- buildings with at least one active operator ritual
- buildings with current diagnostic or proof assets attached
- buildings with timeline activity in the last 30 days
- buildings with structured blockers or obligations instead of document-only state

### 2. Procedural clarity metrics

Examples:

- procedures with explicit active step
- authority requests with tracked response state
- blockers surfaced before work start
- reduction in late-stage complement surprises

### 3. Proof reuse metrics

Examples:

- packs generated from reused proof instead of rebuilt evidence
- proof deliveries linked to prior source artifacts
- scenarios where the same proof served multiple audiences
- reused diagnostic publications in later procedures or packs

### 4. Operating inbox metrics

Examples:

- daily or weekly ControlTower usage
- actions resolved from the queue
- overdue items reduced through routing
- portfolio managers using one queue across buildings

### 5. Adoption and spread metrics

Examples:

- additional actors invited into the same building workspace
- external viewers or pack recipients converted into participants
- repeat pack usage across audiences
- account expansion triggered by workflow success

## Anti-metrics

Do not optimize for:

- raw document count
- raw page views
- raw notifications sent
- raw rule count
- dashboard vanity usage

These can rise while product value stays flat.

## Instrumentation rules

- instrument workflows, not just clicks
- prefer event completion over screen exposure
- tie metrics back to canonical building and source objects
- preserve audience and workflow context

## Decision rule

If a feature cannot improve one of the north-star branches, its priority should
drop.

## Acceptance

This tree is succeeding when the team can answer:

- are buildings becoming more active, not just more populated
- are blockers being surfaced earlier
- is proof being reused more often
- are recurring habits forming
- is multi-actor spread increasing through actual workflow value
