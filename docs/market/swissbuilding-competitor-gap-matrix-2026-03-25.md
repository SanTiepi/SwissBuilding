# SwissBuilding Competitor Gap Matrix

Date de controle: `25 mars 2026`

## Purpose

Cette matrice repond a une question pratique:

`Ou les produits existants sont-ils deja tres bons, et ou SwissBuilding doit-il etre 10x meilleur pour meriter d'exister ?`

La grille n'est pas "plus de features".
La grille est:

- ce qu'ils font deja bien
- ce qu'il ne faut pas recopier
- ou ils laissent de la friction structurelle
- quel resultat SwissBuilding doit produire a la place

## Matrix

| Category | Players | They already do well | Do not try to beat them at | Structural gap left open | SwissBuilding must be 10x better at |
|---|---|---|---|---|---|
| Construction collaboration / CDE | `smino`, `PlanRadar`, `Dalux`, `Procore` | tasks, defects, site capture, plans, BIM views, revision history, field-office collaboration | generic chantier collaboration | building memory stops at project boundary; poor owner or authority continuity; weak procedural readiness | cross-project building memory, procedure readiness, authority-grade packs, reusable proof by audience |
| Swiss property ERP | `GARAIO REM`, `W&W`, `AbaImmo` | contracts, accounting, fees, rent flows, operational back office | ledger or accounting depth | poor dossier clarity across third parties; weak evidence chain; weak procedural visibility | canonical building workspace, proof reuse, deadlines and blockers, multi-actor shared file |
| Portfolio intelligence | `Wuest Dimensions`, `Scandens`, `Optiml` | planning, scenario analysis, decarbonization, portfolio steering | abstract analytics alone | analysis often detached from dossier proof and live procedures | analytics grounded in actual proof, obligations, procedures, blockers, and reusable packs |
| Homeowner passport / helper | `Houzy`, `CLEA`, `Woningpas`, `Gebouwenpas` | simplicity, homeowner language, maintenance reminders, document centralization | consumer-facing convenience alone | not deep enough on Swiss procedures, multi-actor coordination, authority workflow | owner clarity plus institutional-grade procedure, proof, and actor coordination |
| Generic DMS / shared drive | drives, SharePoint-like stacks | storage, permissions, file sync | generic storage | files are dead objects without procedural meaning or reuse discipline | explainable proof, versioned delivery, audience-specific packs, action derived from evidence |
| Specialist diagnostic software | vertical expert tools | expert production, diagnostics, specialized outputs | full vertical execution if someone else already owns it | outputs stay trapped in expert silos | absorb specialist outputs into the building operating file and keep them reusable over time |

## What not to build as primary category

SwissBuilding should not primarily position itself as:

- a chantier task tool
- a Swiss accounting ERP
- a portfolio planning dashboard
- a homeowner content portal
- a generic DMS

Those can exist around the product.
They cannot be the main identity.

## The real competitive wedge

SwissBuilding wins only if it becomes:

- `one building = one canonical workspace`
- `documented` vs `procedurally ready`
- `once-only evidence reuse`
- `one operating inbox`
- `authority-grade workflow and proof`
- `Swiss official-data depth`

## Product tests

If a future module does not improve at least one of these, it is probably not
part of the wedge:

- faster understanding of a building
- fewer repeated document requests
- clearer blocker visibility
- cleaner authority interactions
- more reusable proof across audiences
- stronger continuity across time and actors

## Build / integrate signals

Build deeply:

- procedure engine
- proof delivery and trust
- SwissRules
- operating inbox
- building workspace

Integrate or overlay:

- ERP ledgers
- chantier task tooling
- BIM authoring
- generic document storage
- standalone planning analytics
