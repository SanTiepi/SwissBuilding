# SwissBuilding - Ultimate Product Roadmap

This file is the source of truth for the long-range product trajectory.
SwissBuilding is not positioned as a simple diagnostics platform. It is positioned as an **Agent OS for at-risk buildings**: a system that understands the building, links evidence, explains risk, proposes actions, orchestrates actors, prepares deliverables, and steers the portfolio.

This roadmap should be read together with `docs/vision-100x-master-brief.md`:
- the master brief defines the long-term category and market ambition
- this roadmap defines the execution structure
- `docs/product-frontier-map.md` captures the widest known idea space, uncovered uses, engines, internal tools, and moonshots

## Product Thesis

SwissBuilding must make five things possible:
- the building **understands itself**
- the building **explains itself**
- the building **can be operated**
- the building **can be prepared for intervention**
- the portfolio **can be arbitrated**

The differentiation does not come from a generic chatbot or a nice dashboard.
It comes from the combination of:
- evidence
- memory
- orchestration
- portfolio steering
- controlled agency

Internal ambition rule:
- no artificial ceiling on the product vision
- think Europe-wide category, Swiss launch, local execution layers
- keep the roadmap expansive enough to include future system-defining layers, not just near-term workflow modules
- hold an international-class quality bar:
  - Europe-ready rules and data model
  - multilingual and multi-actor by design
  - strong proof, auditability, exportability, interoperability, and reliability
  - visible UX disciplined enough to stay credible while the hidden system grows

Initial market wedge:
- pollutant diagnostics before renovation
- chosen because it creates the strongest early combination of legal pressure, budget, fragmentation, and data acquisition
- commercial sequencing:
  - Europe-shaped category ambition
  - Switzerland-first launch narrative
  - VD/GE-first execution
  - amiante-first before multi-pollutant expansion
  - safe-to-start dossier overlay above ERP/document systems

## Moonshot Horizon

Beyond the 3 mega-programs, the product should deliberately aim at a more radical end-state.

Target system layers:
- **Building Passport**
  - persistent, versioned, reusable building truth
- **Evidence Graph**
  - proof-linked scores, recommendations, obligations, and interventions
- **Readiness Engine**
  - `safe_to_start`, `safe_to_renovate`, `safe_to_sell`, `safe_to_insure`, `safe_to_intervene`

## Rupture Layers

The roadmap is structured around 5 product layers. The 3 mega-programs deliver them progressively.

### 1. Evidence OS

Purpose:
- make every recommendation provable

Core ideas:
- evidence graph
- explainable risk
- confidence and uncertainty
- source traceability
- auditability

### 2. Building Memory OS

Purpose:
- turn each building into a living, versioned memory rather than a static record

Core ideas:
- persistent digital twin
- source history
- document history
- diagnostic history
- conflicts
- re-qualification
- state over time

### 3. Action OS

Purpose:
- transform risk, missing data, and obligations into orchestrated work

Core ideas:
- manual and system actions
- regulatory playbooks
- campaigns
- assignments
- intervention preparation

### 4. Portfolio OS

Purpose:
- let a manager arbitrate risk, timing, budget, and campaigns across many buildings

Core ideas:
- prioritization
- CAPEX planning
- saved views
- map decision support
- scenario comparison
- national-scale intelligence accumulation over time

### 5. Agent OS

Purpose:
- let specialized invisible agents read, reconcile, detect, suggest, package, and relaunch work across the platform

Core ideas:
- agents embedded in workflows
- no opaque AI magic
- copilote visible only after workflow value is already proven

## Planning Rules

- finish and deepen the operational workflow before opening scale-heavy portfolio work
- keep the critical path smooth: `login -> dashboard -> buildings -> diagnostics -> documents -> simulation -> map`
- anticipate low-regret future domain objects, states, and engines early when that avoids future retrofit debt
- it is easier to hide or delay UI exposure than to retrofit a weak domain model later
- therefore, the plan should think wider than the currently exposed interface
- every major capability must come with:
  - seeded scenarios
  - targeted tests
  - clear permissions
  - complete loading / empty / error / success states
- keep mock validation and real validation explicitly separated
- do not mark a mega-program done unless code, tests, seeded data, and docs all agree

## 48-Month Horizon

The roadmap should think beyond the current program stack. The goal is not only to ship the next major features, but to compound toward a European reference layer.

### Horizon 0-12 months

Focus:
- complete the living and actionable building dossier
- complete multi-actor orchestration and search
- make trust, readiness, and post-works primitives first-class
- turn portfolio from observation into execution

Expected shape:
- a strong Swiss wedge
- evidence-backed workflows
- reusable building memory
- first meaningful portfolio steering

### Horizon 12-24 months

Focus:
- standardize the Building Passport
- make proof packs and handoff packs operational at scale
- deepen campaign logic and portfolio opportunity logic
- strengthen partner ecosystem and interoperability surfaces

Expected shape:
- Swiss reference workflow for pre-work and post-work evidence
- dossier, readiness, and trust becoming reusable operating assets
- first clear market pull beyond the initial wedge

### Horizon 24-36 months

Focus:
- Europe-ready rules expansion
- country packs above the Swiss foundation
- insurer / lender / due-diligence readiness surfaces
- stronger partner APIs, export standards, and ecosystem hooks

Expected shape:
- product no longer feels like a Swiss point solution
- it starts to behave like a cross-jurisdiction operating evidence layer
- the building passport becomes a transferable commercial artifact

### Horizon 36-48 months

Focus:
- reference-system status
- evidence exchange standardization
- building memory transfer between actors
- portfolio intelligence at institutional scale
- embedded agents that coordinate, reconcile, and package work with strong auditability

Expected shape:
- SwissBuilding becomes infrastructure-like
- other systems increasingly depend on its proof, readiness, and memory layers
- the market starts to treat the dossier, trust state, and passport as obvious primitives

### Horizon Rule

- the 48-month horizon is intentionally oversized
- it should guide architecture and object design, not force premature UI exposure
- near-term execution still flows through the active mega-programs and the 60-day runway in `ORCHESTRATOR.md`
