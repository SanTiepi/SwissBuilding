# Source-of-Truth Boundary Matrix

Date de controle: `25 mars 2026`

## Purpose

As SwissBuilding grows, source-of-truth boundaries must stay explicit.

If boundaries blur, duplication and trust loss follow quickly.

## Rule

Every important object needs a clearly named home truth and a clearly named
projection path elsewhere.

## Core boundaries

### Building identity

Home truth:

- SwissBuilding

Projected into:

- packs
- procedures
- imports
- cross-product matching

### Diagnostic execution

Home truth:

- Batiscan

Projected into:

- diagnostic publication
- building memory
- packs
- blockers

### Building procedure state

Home truth:

- SwissBuilding

Projected into:

- ControlTower
- timeline
- packs

### Rule applicability

Home truth:

- SwissRules

Projected into:

- permit_tracking
- procedures
- obligations
- review queue

### Deadlines

Home truth:

- Obligation

Projected into:

- ControlTower
- timeline
- portfolio views

### Proof delivery state

Home truth:

- ProofDelivery and later trust layer

Projected into:

- packs
- timeline
- external audience surfaces

## Product rule

If a feature needs the same truth twice, prefer projection over duplication.

## Acceptance

The matrix is succeeding when the team can say:

- where truth lives
- where it is projected
- who is allowed to change it
