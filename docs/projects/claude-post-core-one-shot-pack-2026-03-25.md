## Internal execution order

### Phase A - Regulatory watch and local honesty

Build the next trust layer on real sources:

- `SwissRules Watch`
- source snapshots
- change events
- review gates
- pilot commune adapter profiles
- commune-level override and variant handling

This phase should ensure:

- freshness is explicit on watched sources
- local variability is surfaced honestly
- commune adapters can fall back to manual review

### Phase B - Exchange and contract backbone

Make SwissBuilding exchangeable and externally legible:

- versioned exchange contracts
- outbound passport publications
- inbound import receipts
- bounded partner capability profiles
- partner API contract families

Rule:

- bounded contract discipline first
- ecosystem complexity later

### Phase C - Trust signals

Turn procedure and proof into trust:

- partner trust signals
- procurement evidence ladder
- public-authority trust signals
- route clarity
- proof completeness
- response traceability
- version discipline

### Phase D - Pilot and demo proof

Make the product easier to prove and sell:

- canonical demo scenarios
- ROI proof calculator grounded in workflow events
- pilot design templates
- pilot scorecards and exit states
- case-study proof story templates

### Phase E - Final coherence pass

Before calling the block done, make sure:

- regulatory freshness influences confidence correctly
- commune-specific caveats reach trust surfaces
- exchange publications and receipts align with proof lifecycle
- demo and pilot proof are grounded in real seeded flows

## Acceptance story

This one-shot is successful when one seeded pilot building can demonstrate all
of this:

1. rules and procedural expectations are tied to explicit source freshness
2. commune-specific overrides or review requirements are visible honestly
3. a passport or pack can be published under a versioned exchange contract
4. an import receipt or downstream acknowledgement can be traced
5. partner or authority trust signals are visible and explainable
6. one demo runbook and one pilot scorecard prove workflow lift with evidence
7. one proof story can be turned into a buyer-legible case study

## Validation discipline

### During implementation

- use changed-file loops only
- prefer targeted backend contract and service tests
- prefer targeted frontend surface tests
- use seeded scenarios over synthetic stubs when possible

### Before closeout

Run:

- `cd backend && ruff check app/ tests/`
- `cd backend && ruff format --check app/ tests/`
- `cd backend && python scripts/run_local_test_loop.py changed`
- `cd backend && python scripts/run_local_test_loop.py confidence`
- `cd frontend && npm run validate`
- `cd frontend && npm run test:changed:strict`

Optional, if environment is ready:

- one real acceptance path covering rule freshness -> commune caveat ->
  exchange publication -> trust signal -> demo proof

Full backend or frontend suite:

- only once at the end
- only after targeted loops are green
