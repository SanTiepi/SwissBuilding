## Summary

Moonshot V2 is no longer a bigger vision deck.
It is a hard benchmark for the next capacity test.

The active question is no longer:

- how many ideas can Claude touch?
- how many layers can be named?
- how many months of human work does the prompt appear to represent?

The new V2 question is:

- how much **infrastructure volume** can Claude actually close
- under a **closure-first** quality bar
- without inflating progress through doc-only, scaffold-only, or surface-only work?

V2 assumes Moonshot V1 has already established the doctrinal and product spine:

- canonical roots
- `BuildingCase` operating root
- `ritual_service` transition discipline
- 5-hub shell
- projection discipline
- source/procedure/document foundation

V2 does not reopen those debates.
It measures how far SwissBuilding can be pushed toward infrastructure-grade closure.

---

## Active Unit of Measurement

The active unit is:

- `closed infrastructure volume`

Time remains telemetry only.
Raw breadth remains secondary.

A capability counts only if it is:

- implemented
- integrated
- tested
- converged into canonical surfaces
- measured with explicit debt and closure evidence

Anything less may still be tracked, but it scores weakly.

## Activation Rule

Activate Moonshot V2 only after Moonshot V1 is explicitly treated as complete enough to stop being the active mission.

- V1 is accepted as closed or hard-cut with explicit accepted debt
- V1 closeout no longer depends on open doctrinal debates

If those conditions are not met:

- continue V1

## The 6 Benchmark Rails

### Rail 1 — Canonical Integrity

Measures whether SwissBuilding preserves one canonical system while expanding.

Required capability areas:

- no parallel truth or role-specific truth copies
- legacy retirement or bounded compatibility with explicit retirement plans
- shell convergence toward master hubs and contextual workspaces
- projection discipline
- ritual discipline
- temporal semantics
- sovereignty boundaries

Closed examples:

- a compatibility bridge with retirement plan, tests, and bounded scope
- shell contraction that removes duplicated surfaces
- projection registry enforcement that prevents page-owned business truth

### Rail 2 — Infrastructure Excellence

Measures whether the repo itself becomes infrastructure-grade.

Required capability areas:

- CI hardening
- repo fitness functions
- dependency and vulnerability governance
- schema/API registry discipline
- migration/backfill safety
- release/environment parity
- observability and rollback posture
- repo excellence guardrails

Closed examples:

- machine-checkable route-shell drift guard
- tracked-artifact guard in CI
- migration safety checks that prevent silent destructive drift

### Rail 3 — Source and Connector Reliability

Measures whether source depth becomes operationally reliable, not just connected.

Required capability areas:

- source registry completeness
- source health events
- incident handling
- schema drift detection
- fallback chains
- freshness/watch contracts
- degraded-mode visibility in canonical workspaces

Closed examples:

- a source adapter with health, fallback, tests, and workspace effects
- explicit connector degradation shown as source unavailability instead of fake absence of truth

### Rail 4 — Exchange and Conformance

Measures whether SwissBuilding behaves like a reliable exchange substrate.

Required capability areas:

- truth API contracts
- exchange manifests
- import/export validation
- conformance engine
- rights/license/externalization governance
- multi-actor share/transfer boundaries
- bounded re-import symmetry

Closed examples:

- contract-tested API endpoints with stable shapes
- machine-readable manifest validation for a passport or pack exchange flow

### Rail 5 — Procedure and Operational Depth

Measures whether procedure and operations become deeply executable, not just modeled.

Required capability areas:

- procedure grammar closure
- work-family coverage with real operational meaning
- `BuildingCase`-attached execution convergence
- invalidation propagation
- Trust Ops and review machine
- operational reaction loops
- consequence and blocker propagation

Closed examples:

- procedure deltas that reopen blockers, invalidate packs, and surface review tasks
- work-family depth that changes actual forms, packs, or next actions

### Rail 6 — Market Infrastructure Readiness

Measures whether SwissBuilding starts to act like a governed market substrate.

Required capability areas:

- partner APIs
- contributor quality
- partner trust
- exchange protocols
- bounded autonomous coordination
- network-grade interoperability
- explainable external reliance

Closed examples:

- partner-facing API or exchange contract with explicit governance and compatibility rules
- agentic coordination that remains auditable and reversible

## Capability Unit Model

Moonshot V2 is scored through `CapabilityUnit`s, not narrative blocks.

### Capability Unit Fields

Every tracked unit must declare:

- `name`
- `rail`
- `status`
- `quality_gates`
- `evidence`
- `canonical_surfaces_touched`
- `legacy_surfaces_reduced`
- `debt_classification`
- `next_closure_step`

### Capability Status

Each unit gets exactly one status:

- `closed`
- `partial`
- `compat-only`
- `doc-only`
- `not-started`

### Quality Gates

Each unit has 5 binary gates:

- `implemented`
- `integrated`
- `tested`
- `converged`
- `operationalized`

Rule:

- a unit may not be marked `closed` unless all 5 gates are true

### Scoring Weights

- `closed` = `1.0`
- `partial` = `0.35`
- `compat-only` = `0.2`
- `doc-only` = `0.05`
- `not-started` = `0`

V2 progress should be expressed as:

- per-unit status
- per-rail closure summary
- aggregate `BenchmarkScore`
- current `BenchmarkGrade`

## What Counts as Weak or Zero Real Credit

These may be tracked, but do not meaningfully advance the benchmark on their own:

- doc-only layers
- new routes without shell convergence
- new models/services without canonical destination
- compatibility bridges without retirement plan
- APIs without contract tests
- seeds without repeatable validation
- extractors without consequence wiring
- dashboards without operational reactions
- pages that duplicate master workspaces

Practical rule:

- work that is visually impressive but not converged or operationalized should score weakly
- repo-grade hardening with machine-checkable enforcement does count as strong progress

## Benchmark Grades

### Grade D — Wide but Soft

- many partials
- little real closure
- too much doc/scaffold progress
- too much drift hidden by breadth

### Grade C — Strong Nucleus

- canonical core is strong
- shell/API/source/procedure foundation is real
- but outer infrastructure remains fragile or uneven

### Grade B — Infrastructure-Grade System

- repo excellence, connector reliability, conformance, migration safety, and exchange discipline are materially closed
- the system is broad and structurally durable

### Grade A — Market Infrastructure Candidate

- the system is broad, deep, externally governable, measurable, durable, and interoperable
- SwissBuilding behaves like infrastructure, not just a strong application

Recommended V2 success target:

- minimum: `Grade B`
- stretch: `Grade A`

## Acceptance Rules

Moonshot V2 is only successful if the benchmark can distinguish:

- impressive breadth
- true closure
- infrastructure maturity

The benchmark is only valid if:

- every major V2 area can be expressed as a `CapabilityUnit`
- every unit uses binary quality gates
- repo excellence counts as benchmark volume
- doc-only progress is explicitly devalued
- shell convergence and legacy reduction are measured
- another reviewer can score V2 without rereading the full strategy stack
- another agent can continue V2 without reopening what `done` means

Representative acceptance scenarios:

- a new source adapter with no health/fallback/tests cannot score as `closed`
- a new page that duplicates a master workspace cannot score as meaningful progress
- a new API without contract tests cannot score as `closed`
- a compatibility bridge with no retirement plan cannot score above `compat-only`
- a repo-quality improvement with machine-checkable enforcement does score as real V2 volume
