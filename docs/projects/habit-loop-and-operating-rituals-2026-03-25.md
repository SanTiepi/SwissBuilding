# Habit Loop and Operating Rituals

Date de controle: `25 mars 2026`

## Purpose

SwissBuilding should not just be useful.

It should create durable operating habits.

This document defines the repeatable rituals that make the product sticky:

- daily
- weekly
- event-driven

## Rule

A surface is strategically stronger when it supports a recurring habit rather
than a one-off consultation.

## Daily rituals

### Property manager

Open:

- `ControlTower`

Need to answer:

- what is overdue
- what is blocked
- what needs routing today
- which building needs attention first

Win condition:

- one queue replaces scattered reminders, inboxes, and memory

### Building owner

Open:

- building overview

Need to answer:

- what changed
- what is risky
- what is next
- what evidence exists

Win condition:

- confidence without hunting through documents

### Procedure coordinator

Open:

- procedure view
- complement requests

Need to answer:

- what step is active
- what is missing
- what must be sent
- what is waiting on authority

Win condition:

- procedure is managed in-product, not in email

## Weekly rituals

### Portfolio review

Open:

- portfolio ControlTower
- benchmarking grounded in proof

Need to answer:

- where the biggest blocker density sits
- which obligations are becoming systemic risk
- where a wave or campaign should be launched

### Proof and readiness review

Open:

- proof reuse moments
- pack readiness surfaces

Need to answer:

- which buildings are reusable-proof rich
- which packs still need missing pieces
- where evidence exists but is not yet leveraged

### Regulatory watch review

Open:

- SwissRules watch
- manual review queue

Need to answer:

- what changed
- which building clusters are affected
- which cases now need review

## Event-driven rituals

### New document arrives

Reflex:

- classify
- attach
- decide whether it changes blockers, proof, or obligations

### New diagnostic publication arrives

Reflex:

- link to building
- refresh proof reuse opportunities
- surface any changed blocker or pack readiness

### Authority request arrives

Reflex:

- attach to procedure
- create or update deadline
- route responsibility
- expose missing proof explicitly

### Ownership or manager changes

Reflex:

- generate handoff package
- preserve unresolved blockers and caveats
- create next-operator continuity

## Habit design rules

- every recurring ritual must start from one primary surface
- every ritual must end with a next action, not just information
- every ritual should reduce the need for a second tool
- every ritual should reinforce canonical building truth

## Anti-pattern

If a surface is important but does not fit into:

- a daily ritual
- a weekly ritual
- or a strong event-driven ritual

it is at risk of becoming dead UI.

## Acceptance

SwissBuilding is winning when users can say:

- "this is where I start my day"
- "this is where I check what matters this week"
- "this is where I handle surprises when they happen"
